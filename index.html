First Principles

Why AIHub exists

The token cost problem is a structural tax on intelligence

Every time an AI agent browses the web — parsing raw HTML, OCR-ing rendered pages, re-summarising noise — it burns tokens on work that should never reach the model. The web was built for humans. AI gets the worst of it.

AIHub's answer is simple: intercept at the source. Browser extensions sit inside the page's own JavaScript execution context. They see the same structured data the app renders — perfectly parsed, token-efficient, already typed.

A local bridge app converts that structured stream into a clean REST API. AI agents consume structured JSON, not DOM soup. The result: richer context, lower cost, more reliable automation.

// Without AIHub — raw page scraping
tokens_used = 28 400
// Full HTML + images + noise
// Re-parsed every single call

// With AIHub — structured bridge
tokens_used = 1 200
// Clean JSON from extension
// Already structured, typed, minimal

// Saving: ~96% fewer tokens
// Speed: ~8× faster per query

A platform, not just a tool

AIHub is designed to be the universal hub layer. Any software — yours or a third party's — can join the protocol and become instantly accessible to AI agents.

⬡ The AIHub Protocol

Any application that implements the AIHub WebSocket handshake can register itself with LocalBridge and expose its capabilities as REST endpoints. Extensions plug in and out at runtime — no restarts, no configuration files.

Integration Guide for AI Agents

This section is written for AI agents (like ClawBot) accessing AIHub on behalf of their users. Follow these instructions to understand available capabilities and execute tasks autonomously.

⚠ Prerequisites — verify before proceeding
01
User is logged into X (Twitter)

The owner of this session must be signed in to X in the same Chrome profile where TweetClaw is installed. Without an active session all Twitter API calls will fail.

02
TweetClaw Chrome extension is installed & active

The extension must be running and connected. It acts as the data bridge between the Twitter web app and LocalBridge.

03
LocalBridge desktop app is running

The companion app must be open and listening on localhost. Default port is 10088. If you receive a connection refused error, ask the user to launch LocalBridge first.

Step 1 — Capability Discovery

Fetch the complete, self-describing API documentation from the running LocalBridge instance. This endpoint returns all available routes, parameters, and response schemas in a format you can parse and reason over directly — no hardcoded knowledge required. Always call this first; the docs are the authoritative source of truth.

shell # Discover every capability provided by TweetClaw + LocalBridge 
curl -X GET http://127.0.0.1:10088/api/v1/docs
→ Parse the returned JSON to understand every available endpoint before making task-specific calls. Prefer the live docs over any cached or assumed knowledge.
Step 2 — Autonomous Execution

Once you have the API map, compose and execute calls based on your owner's instructions. Current coverage is X / Twitter. Combine endpoints freely to fulfil complex tasks.

Twitter / X
  • Read home timeline & search results
  • Fetch tweet detail & thread replies
  • Look up user profiles by screen name
  • Post tweets, replies, and quotes
  • Like, retweet, bookmark, follow, unfollow
  • Open, navigate & close browser tabs

Error Handling

All error responses include a structured { code, message } object. Common codes and recommended recovery actions:

NOT_CONNECTED LocalBridge is not running — ask the user to start the app.
SESSION_INVALID Twitter session expired — ask the user to log back into X.
NO_X_TAB No open X tab found — call the open_tab endpoint first, then retry.
DATA_NOT_CAPTURED Navigate to the relevant X page, wait for it to load, then retry.
